## The game

Tic-tac-toe is played on a 3×3 grid. Two players take turns placing their mark in an empty cell. The first to complete a row, column, or diagonal wins. If the board fills with no winner, the game is a draw.

- **Episode** — one complete game from the opening move to a terminal outcome.

## Value learning

<BookReading
  href="http://incompleteideas.net/book/the-book.html"
  bookTitle="Reinforcement Learning: An Introduction"
  authors="Sutton & Barto"
  chapter={1}
  section="Section 1.5"
/>

Value learning works by estimating how good each state is under the agent's current policy. We store a **tabular value function** — one number `V(s)` per board configuration.

After each game, we apply an update to every state the agent visited:

<Math
  display
  latex="V(s) \leftarrow V(s) + \alpha \bigl(R - V(s)\bigr)"
/>

where `R` is the terminal reward and `α` is the learning rate.

Let's try and understand the above update rule in the context of the current problem. We will take one state and think about what the above rule would do for it.
Let's say the agent reaches that state in a game. After few more moves, if the agent wins the game, then we would think that this was
a good state to be in. Since the value of the state represents how good is a state, we would want to increase the value for this state.
On winning, the reward is +1. What we want is to increase the value of the state so that it is closer to this reward. Hence we take the
difference between the reward and the current value and then increment the current value by a step which is proportional to the delta.
If the current value of the state is 0.2, then there is a large difference between it and the target value of 1, and hence we want to take
a larger step. The proportionality factor is `α`. Similarly, if we lose a game from that state, we would want to reduce the value of 
the state so that it moves closer to -1, the reward the agent gets when it loses. 

The agent chooses moves with an **ε-greedy policy**: with probability ε it plays a random legal move; otherwise it picks the move whose **successor state** has the highest stored value.
We need to use a policy which sometimes plays random legal moves so that it can explore different positions and learn which positions
might be good.

### Player

Both the learning agent and the opponents are **players**. `game.py` defines a common interface they all implement:

<CodeBlock lang="python" title="game.py — Player" code={playerSnippet} />

Every player is constructed with a **symbol** — `−1` for O or `+1` for X. That symbol is the mark the player places on the board when it is their turn.

`move` is the only method subclasses must implement. The game loop calls `move(game)` on the current player; the player reads the board from `game`, chooses a position, and updates the board by calling `game.set_position`.

`on_result` is optional. It is called once when the game ends, with the final winner (`−1`, `0` for a draw, or `+1`). The default implementation does nothing. Our learning **Agent** overrides it to run the value update; scripted opponents leave it unchanged.

This split keeps the game rules in `TicTacToe` and puts all decision-making in `Player` subclasses — whether that is a fixed rule, random play, minimax, or a learned policy.

### Why performance can be poor with few training games

With very few games, most board indices are never visited. ε-greedy exploration means early episodes are noisy. The value table stays near zero, so greedy play is effectively arbitrary — wins against a weak opponent are not yet reliable. As training games accumulate, the agent revisits the same lines repeatedly and values converge for the states on those lines.

## Part 3 — Agent vs agent

Finally, both sides are learning **Agent** instances with the same ε-greedy policy. Each player updates its own value table during training.

- The training opponent is exploratory (ε-greedy), not fixed.
- As both improve, each faces progressively stronger — but still stochastic — opposition.
- Neither player is optimal, but co-evolution exposes both to rich game trajectories.

## Why more training games help

Across all three parts, low training budgets hurt for the same underlying reasons:

- **Sparse coverage** — most of the 19,683 states are never visited in a handful of games.
- **Noisy updates** — ε-greedy exploration and random opponents inject variance into which states get updated and by how much.
- **Delayed signal** — reward arrives only at episode end, so early value estimates are rough until similar trajectories repeat.
- **Opponent-specific learning** — against a fixed weak opponent, extra training helps only until the policy saturates for that opponent; it does not automatically transfer to a different one.

More training games let the agent revisit important states, average out exploration noise, and — when the opponent is diverse enough — build values that transfer to stronger held-out opponents like **OptimalOpponent**.

## Summary

| Idea | Takeaway |
|------|----------|
| State values | One number per board; greedy play uses `V(s')` of successor states |
| Monte Carlo update | Terminal reward backs up to all visited states |
| Opponent = environment | Good eval against the training opponent ≠ good play in general |
| Fixed opening | Opponent-first with deterministic O fixes the first move — limits what X learns |
| Exploration | Random and self-play opponents widen the training distribution |
| Evaluation | Always test against held-out opponents; score = wins − losses over 10 games |
